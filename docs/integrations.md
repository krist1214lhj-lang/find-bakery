# 외부 연동 계약

작성 기준일: 2026-06-22
상태: 카카오 장소 검색 서버 경계 구현

외부 서비스의 응답, 쿼터, 약관은 변경될 수 있다. 구현과 배포 직전에 공식 문서를 다시 확인하고 확인일을 이 문서에 갱신한다.

## 1. 공통 경계 원칙

```text
외부 서비스
  → 공급자별 클라이언트
  → 런타임 응답 검증
  → 내부 정규화 모델
  → 도메인 서비스
  → 데이터베이스 또는 화면
```

- 화면과 도메인 로직에서 공급자 원본 응답을 직접 사용하지 않는다.
- 외부 ID는 `provider + externalId` 조합으로 별도 저장한다.
- 네트워크 오류, 시간 초과, 쿼터 초과, 빈 응답을 구분한다.
- 실패한 외부 조회가 기존의 검증된 값을 자동 삭제하거나 덮어쓰지 않게 한다.
- 재시도는 멱등 조회에만 제한적으로 적용하고 지수 백오프를 사용한다.
- API 키와 서비스 역할 키는 서버 전용 환경 변수로 관리한다.
- 로그에 키, 사용자 위치 원문, 제보 자유 입력을 기록하지 않는다.

## 2. 카카오 로컬 API

공식 문서: <https://developers.kakao.com/docs/latest/ko/local/dev-guide>

용도:

- 주소를 WGS84 좌표로 변환
- 키워드로 장소 검색
- 장소의 이름, 주소, 전화번호, 좌표와 카카오맵 상세 URL을 교차 확인

공식 문서상 REST API 키를 `Authorization: KakaoAK ...` 헤더에 담아 요청한다. 키워드 검색은 위치, 반경, 정렬, 페이징 조건을 지원한다.

내부 어댑터 계약:

```ts
type PlaceProvider = {
  searchPlaces(input: PlaceSearchInput): Promise<PlaceSearchResult>;
  geocodeAddress(address: string): Promise<GeocodeCandidate[]>;
};
```

현재 구현:

- `/api/places/search`에서만 카카오 REST API를 호출
- 지역·상호 키워드 검색과 현재 위치 기준 거리순 검색 지원
- 지역명만 입력하면 `빵집` 검색 의도를 보강해 비관련 장소 혼입 축소
- 5초 시간 제한, 인증·쿼터·빈 결과·잘못된 응답 오류 분리
- 카카오 원본 응답을 내부 `PlaceCandidate` 모델로 변환
- 카카오 결과를 검증된 `Bakery` 데이터와 섞지 않고 `미검증 후보`로 표시
- 사용자가 선택한 후보만 서버 서명 토큰을 검증해 검수 큐에 저장
- 승인 전 후보는 공개 상세·영업 상태·정식 저장 대상에서 제외
- 관리자 승인 시 지도 출처와 C등급 확인 기록을 남기고 `확인 필요` 지점으로 등록
- 사용자 위치는 요청에만 사용하며 로그·DB·분석 도구에 저장하지 않음
- 실제 호출에는 서버 전용 `KAKAO_REST_API_KEY` 설정 필요
- 2026-06-22 `성수동` 실제 검색에서 제과·베이커리 후보 15곳 응답 확인

정규화 시 주의:

- 응답 좌표 문자열을 유효 범위의 숫자로 변환
- 도로명 주소가 없을 수 있으므로 지번 주소와 구분
- 전화번호는 비교용 정규화 값과 표시 값을 분리
- 같은 상호명이 여러 지점에 존재한다고 가정
- 검색 결과는 사실 확정이 아닌 `Source`와 검증 후보로 저장

환경 변수:

- `KAKAO_REST_API_KEY` — 서버 전용
- `NEXT_PUBLIC_KAKAO_MAP_JS_KEY` — 브라우저 지도 SDK 도입 시 공개 범위를 검토한 뒤 사용

배포 전 확인:

- 현재 쿼터와 과금 정책
- 지도 표시 및 데이터 저장에 관한 이용 조건
- 허용 도메인과 앱 설정
- 출처 표시 요구사항

## 3. 소상공인시장진흥공단 상가(상권)정보 API

공식 문서: <https://www.data.go.kr/data/15012005/openapi.do>

용도:

- 전국 영업 상가업소의 상호명, 업종, 주소, 좌표 교차 확인
- 신규 후보 발굴과 폐점·변경 검수 보조

공공데이터포털에 표시된 정보 기준으로 REST, JSON/XML 형식을 지원한다. 업데이트 주기와 트래픽 한도는 배포 전 재확인한다.

2026-06-22 공식 포털 재확인:

- 페이지 수정일: 2026-04-20
- 개발계정 트래픽: 10,000건
- 개발·운영 활용신청: 자동승인
- 이용허락범위: 제한 없음
- 주요 필드: 상가업소번호, 상호명, 업종명, 주소, 경도, 위도

내부 어댑터 계약:

```ts
type StoreRegistryProvider = {
  searchStores(input: StoreRegistrySearchInput): Promise<StoreRegistryResult>;
};
```

정규화 시 주의:

- 업종 코드만으로 유명 빵집 여부를 판단하지 않음
- 영업 중 데이터라는 설명을 폐점 부재의 절대 증거로 사용하지 않음
- 행정구역 코드와 주소 텍스트를 함께 보존
- 좌표계와 숫자 유효성 확인
- 결과는 사용자 화면에 즉시 게시하지 않고 검증 후보로 처리

환경 변수 후보:

- `DATA_GO_KR_SERVICE_KEY` — 서버 전용
- `DATA_GO_KR_STORE_API_URL` — 상세 명세의 반경 조회 URL 변경 시에만 재정의

현재 구현:

- 반경 300m 상가 조회 응답의 런타임 검증과 내부 `StoreRegistryRecord` 변환
- 공식 상권업종 중분류 `I210`(기타 간이)로 서버 조회를 좁히고 `빵/도넛`, `떡/한과` 등 제과 관련 업종만 교차 확인 대상으로 필터
- 카카오 후보와 상호·주소·좌표를 비교해 일치 점수와 근거 산출
- 관리자 장소 후보 화면의 `공공데이터 확인` 동작과 오류 상태
- 고정 fixture 기반 정상·필터·키 누락 단위 테스트
- 일치 결과를 `place_candidate_evidence`에 멱등 저장
- 70점 이상 근거가 있는 후보 승인 시 주소·좌표의 지도·공공데이터 기록을 B등급으로 승격
- B등급 승격과 두 출처 생성 로컬 DB 통합 검증 통과
- 실제 공공 API 승인 전파 후 200 응답 확인
- 공식 상권업종 중분류 `I210` 적용으로 반경 내 제과 관련 결과만 조회
- 카카오 `퍼먼트`와 상가정보 `퍼먼트알티드`가 상호 유사·주소 일치·좌표 1m로 90점 일치
- 실제 일치 근거 저장→승인→B등급·두 출처 생성 통합 검증 통과

## 4. SNS와 공식 채널

대상:

- 공식 홈페이지
- 인스타그램
- 네이버 블로그·플레이스 소식
- 카카오채널
- 기타 매장 공식 공지

MVP 원칙:

- 무단 자동 수집을 기본 전략으로 삼지 않는다.
- 공개 원문을 운영자가 확인하고 URL, 게시일, 적용 기간, 확인 항목을 등록한다.
- 공식 API나 허가된 수집 경로가 없는 플랫폼은 자동화하지 않는다.
- 계정 로그인, 비공개 콘텐츠, CAPTCHA 우회를 요구하는 수집은 수행하지 않는다.
- 원문이 사라져도 검증 이력은 남기되 현재 접근 불가 상태를 표시한다.

추후 자동화 전 확인:

- 플랫폼 공식 API 제공 범위
- 콘텐츠 저장·표시 권리
- 호출량과 인증 요구
- 개인정보와 사용자 생성 콘텐츠 처리
- 삭제·정정 요청 대응

## 5. Supabase

공식 문서: <https://supabase.com/docs/guides/database/overview>

용도:

- PostgreSQL 데이터베이스
- 로그인 사용자 저장·제보
- 행 수준 보안 정책
- 이미지 저장소 도입 가능성

경계:

- 공개 조회와 관리자 쓰기 권한을 분리
- 브라우저에는 공개 가능한 키만 제공
- 서비스 역할 키는 서버 작업에서만 사용
- 모든 공개 테이블에 행 수준 보안 정책을 명시
- 마이그레이션이 데이터베이스 구조의 단일 기준

환경 변수 후보:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SECRET_KEY` — 서버 전용, 새 프로젝트 우선
- `SUPABASE_SERVICE_ROLE_KEY` — 레거시 프로젝트 호환용 서버 전용

현재 구현:

- 홈·탐색·상세 화면은 공개 anon/publishable 키와 RLS로 조회
- 익명 제보는 `/api/reports` Route Handler에서 검증 후 저장
- 서버 설정이 없으면 브라우저와 API가 `local-only` 모드를 협상
- 관리자 검수는 `review_correction_report` RPC로만 상태 변경
- 개발 관리자 큐는 서버 전용 키로 제보를 읽고 RPC로 검수
- 서비스 역할 검수도 `server_service` 감사 주체를 기록
- 2026-06-19 로컬 브라우저 제출과 `correction_reports` 실제 INSERT 확인
- 2026-06-19 관리자 큐 표시, 승인 상태 전이, `review_actions` 생성 확인
- 자세한 적용 절차는 `docs/database-setup.md`

## 6. Vercel

공식 문서: <https://vercel.com/docs/environment-variables>

용도:

- 미리보기와 테스트 배포
- 환경별 환경 변수 관리
- 배포 빌드와 롤백

원칙:

- 개발, 미리보기, 운영 환경 값을 분리
- 환경 변수 이름만 `.env.example`에 기록하고 실제 비밀값은 커밋하지 않음
- 미리보기 환경이 운영 데이터에 쓰지 못하게 분리
- 배포 전 필수 환경 변수 검증 스크립트 실행

## 7. 오류 모델

외부 연동 오류를 다음 공통 코드로 변환한다.

- `PROVIDER_UNAVAILABLE`
- `PROVIDER_TIMEOUT`
- `PROVIDER_RATE_LIMITED`
- `PROVIDER_UNAUTHORIZED`
- `PROVIDER_INVALID_RESPONSE`
- `PROVIDER_EMPTY_RESULT`
- `PROVIDER_TERMS_REVIEW_REQUIRED`

사용자 화면에는 공급자 내부 오류를 그대로 노출하지 않는다. 관리자 로그에는 공급자, 요청 식별자, 발생 시각, 공통 오류 코드만 남기고 민감한 요청 원문은 제외한다.

## 8. 캐시와 갱신 초안

- 사용자 검색: 짧은 캐시, 공급자 약관 범위 내 사용
- 지점 검증 후보: 검수 완료 전 별도 저장
- 공식 검증 정보: 데이터베이스 값을 우선 표시하고 백그라운드 재검토
- 외부 장애 시: 마지막 확인일과 함께 기존 값을 표시
- 임시 일정: 적용 기간 전후로 우선 재확인

정확한 보존 기간은 각 공급자의 최신 이용 조건을 확인한 뒤 결정한다.

## 9. 연동 테스트

각 공급자마다 다음 계약 테스트를 둔다.

- 정상 응답 정규화
- 선택 필드 누락
- 빈 결과
- 잘못된 좌표·날짜·전화번호
- 인증 실패
- 시간 초과
- 쿼터 초과
- 공급자 필드 추가·변경 시 안전한 무시 또는 실패

실제 API를 호출하는 테스트와 고정 fixture를 사용하는 테스트를 분리한다. CI 기본 테스트는 fixture 기반으로 실행하고 실제 호출은 수동 또는 예약 검증으로 제한한다.
