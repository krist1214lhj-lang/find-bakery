# 전국 유명 빵집 탐색 앱 제품 계획

작성 기준일: 2026-06-18

## 1. 제품 목표

전국의 유명하고 개성 있는 빵집을 사용자가 쉽고 즐겁게 발견하도록 돕되, 주소·영업시간·휴무일·대표 메뉴처럼 방문 결정에 중요한 정보는 출처와 확인 시점을 함께 제공하는 신뢰도 높은 빵집 탐색 서비스를 만든다.

핵심 문장:

> 예쁜 빵집 지도가 아니라, 믿고 출발할 수 있는 빵집 지도.

## 2. 핵심 사용자

1. 여행 전에 지역별 유명 빵집을 찾는 빵지순례 사용자
2. 현재 위치 주변에서 실패 가능성이 낮은 빵집을 찾는 사용자
3. 소금빵·베이글·케이크·비건 등 특정 메뉴나 취향으로 찾는 사용자
4. 방문할 빵집을 저장하고 동선을 계획하려는 사용자

## 3. 경쟁 서비스에서 배울 점

### 빵긋

- 장점: 주변·지역별 탐색, 메뉴·가격·사진, 메뉴 단위 리뷰, 인기 메뉴, 저장, 근처 알림
- 보완 기회: 초기 리뷰 수가 적을 때 추천 신뢰도가 약해질 수 있으므로 공식 정보와 검증 상태를 별도 표시해야 함

### 빵지술래

- 장점: 동네 빵집 전문성, 세일 정보, 예약·픽업으로 실제 구매까지 연결
- 보완 기회: 거래 기능은 운영 복잡도가 높고, 전국의 유명 빵집을 발견하는 경험과는 목적이 다름

### 떠리

- 장점: 마감 할인 상품의 실시간 탐색과 예약·픽업이라는 분명한 효용
- 보완 기회: 제휴 매장과 할인 상품 중심이어서 전국 빵집 정보 서비스로는 범위가 제한됨

### 오븐링크

- 장점: 전국 인기 빵집의 택배·픽업 주문, 판매 일정 알림, 메뉴 중심 쇼핑
- 보완 기회: 주문 가능한 입점 매장 중심이므로 방문 탐색과 객관적인 정보 비교가 약할 수 있음

### 백년가게맵

- 장점: 공공데이터 기반, 지도·지역·테마 탐색, 가고 싶은 곳/다녀온 곳 구분, 외부 지도 연결
- 보완 기회: 빵집 특화 메뉴·취향 필터와 최신 영업 정보 검증은 별도 보강이 필요함

### 범용 지도·맛집 앱

- 장점: 방대한 장소 수, 길찾기, 많은 리뷰, 익숙한 사용성
- 보완 기회: 광고·인기순 노출과 정보 과잉 때문에 빵집만 깊이 탐색하기 어렵고, 지점별 대표 빵이나 품절 가능성 같은 도메인 정보가 약함

## 4. 제품 차별화 원칙

### 4.1 정확성을 화면에 보이게 만들기

각 정보에 다음 상태를 둔다.

- 공식 확인: 매장 공식 홈페이지·SNS·전화 등으로 확인
- 교차 확인: 지도 API와 공공데이터 등 2개 이상 출처가 일치
- 사용자 제보: 아직 운영자가 검토하지 않은 변경 정보
- 확인 필요: 일정 기간 이상 갱신되지 않았거나 출처가 충돌

상세 화면에는 `마지막 확인일`, `확인 출처`, `정보 수정 제보`를 노출한다.

### 4.2 빵집보다 ‘먹고 싶은 빵’에서 시작하기

지역뿐 아니라 소금빵, 크루아상, 베이글, 식사빵, 케이크, 전통빵, 비건, 글루텐 프리 등으로 탐색한다.

### 4.3 유명세의 근거를 설명하기

단순 별점 대신 수상·방송·지역 대표성·오랜 영업·전문 메뉴·사용자 저장 수 등 선정 근거를 분리해 보여준다.

### 4.4 감성 디자인과 정보 가독성을 분리하지 않기

따뜻한 크림색, 구움색, 버터색을 기본으로 사용하되 핵심 정보는 높은 대비와 명확한 계층으로 표현한다. 사진은 탐색을 끌어당기고, 텍스트는 방문 판단을 빠르게 끝내야 한다.

## 5. MVP 범위

### 반드시 포함

1. 홈: 지역·테마·대표 메뉴 기반 추천
2. 지도/목록 전환 탐색
3. 검색과 필터
   - 지역
   - 빵 종류
   - 현재 영업 여부
   - 주차·좌석·포장
   - 검증 상태
4. 빵집 상세
   - 지점명, 주소, 연락처
   - 영업시간, 휴무일
   - 대표 메뉴와 가격
   - 편의 정보
   - 공식 링크
   - 마지막 확인일과 출처
5. 가고 싶은 곳 저장
6. 정보 오류·폐점·영업시간 변경 제보
7. 운영자용 데이터 검수 화면

### 첫 배포에서 제외

- 앱 내부 결제·예약·배달
- 자유형 커뮤니티
- 별점 중심 리뷰 시스템
- 복잡한 소셜 팔로우
- AI가 출처 없이 생성하는 추천 문구

정확성을 우선하기 위해 거래와 커뮤니티는 검증 체계가 안정된 뒤 추가한다.

## 6. 데이터 수집·검증 체계

### 기본 데이터

- 상호명과 지점명
- 주소와 좌표
- 전화번호
- 영업시간과 휴무일
- 공식 홈페이지·SNS
- 대표 메뉴, 가격, 메뉴 사진
- 주차·좌석·포장·택배 여부
- 폐점·이전·임시휴무 상태

### 추천 출처 우선순위

1. 매장 공식 홈페이지·공식 SNS·매장 직접 확인
2. 카카오 장소 검색 등 지도 API
3. 소상공인시장진흥공단 상가업소 정보
4. 한국관광공사·지자체·공공데이터
5. 신뢰 가능한 언론·방송·수상 기록
6. 사용자 제보

### SNS 검증 활용

인스타그램, 네이버 블로그·플레이스 소식, 카카오채널 등은 임시휴무, 영업시간 변경, 이전, 당일 메뉴와 품절 안내가 빠르므로 최신 정보 검증의 주요 출처로 활용한다.

SNS 출처는 다음 세 종류로 구분한다.

- 공식 계정: 매장 홈페이지나 지도 서비스에서 연결되었거나, 사업자가 직접 확인한 계정
- 준공식 계정: 대표자·직원 계정으로 보이지만 매장과의 관계가 완전히 확인되지 않은 계정
- 일반 사용자 게시물: 방문 후기, 사진, 댓글 등 매장 운영자가 작성하지 않은 정보

공식 계정의 최신 공지는 영업시간·휴무·이전·메뉴 변경의 근거로 사용할 수 있다. 준공식 계정과 일반 사용자 게시물은 이상 징후를 발견하는 보조 자료로만 사용하고, 단독으로 확정 정보에 반영하지 않는다.

SNS로 확인할 때 다음 정보를 함께 기록한다.

- 플랫폼과 계정 주소
- 공식 계정 확인 여부와 확인 근거
- 원문 게시물 주소
- 게시일과 실제 적용 기간
- 확인한 정보 항목
- 운영자 확인일
- 캡처가 아닌 원문 접근 가능 여부

스토리처럼 사라지는 정보는 단기 공지로만 취급하고, 영구적인 영업 정보 변경에는 별도 교차 확인을 요구한다.

### 검증 규칙

- 영업시간과 휴무일은 공식 출처가 없으면 ‘변동 가능’ 표시
- SNS 게시물은 작성일이 아니라 실제 적용 날짜를 기준으로 반영
- “오늘”, “이번 주”, “내일부터” 같은 표현은 게시일을 기준으로 절대 날짜로 변환해 저장
- 명절·행사·기상 상황에 따른 임시 공지는 상시 영업시간을 덮어쓰지 않고 별도 일정으로 관리
- 공식 SNS와 지도 정보가 충돌하면 최신 공식 공지를 우선 표시하되 ‘지도 정보와 다름’ 상태로 검수
- 댓글과 해시태그만으로 폐점·이전·영업시간 변경을 확정하지 않음
- SNS 팔로워 수나 화제성은 유명세 참고 지표로만 사용하며 정보 정확성 점수에는 반영하지 않음
- 같은 브랜드의 지점을 별도 장소로 관리
- 주소·전화번호·좌표가 충돌하면 자동 게시하지 않고 검수 대기
- 메뉴 가격은 확인 날짜를 반드시 저장
- 모든 수정 이력을 남기고 이전 값으로 되돌릴 수 있게 설계
- 30일/90일/180일 주기로 중요도에 따라 재확인

### 검증 점수 예시

- A — 최근 30일 이내 공식 채널 또는 매장 직접 확인
- B — 최근 90일 이내 공식 채널과 지도·공공데이터가 일치
- C — 출처는 있으나 90일 이상 경과했거나 일부 항목만 확인
- D — 사용자 제보 또는 출처 충돌로 재확인 필요

검증 등급은 빵집 전체에 하나만 부여하지 않고 주소, 영업시간, 휴무일, 메뉴·가격 등 항목별로 관리한다.

## 7. 추천 기술 방향

초기에는 설치 장벽이 낮고 테스트 배포가 빠른 반응형 웹/PWA로 시작한다.

- 프런트엔드: Next.js + TypeScript
- UI: 재사용 가능한 디자인 시스템과 모바일 우선 반응형 구성
- 데이터베이스/인증: Supabase + PostgreSQL
- 위치 검색: Kakao Maps/Local API
- 배포: Vercel
- 이미지: 최적화 저장소 사용
- 운영 도구: 동일 프로젝트 안에 관리자 화면 구성

사용성이 검증되면 PWA를 유지하거나 React Native/Expo 기반 앱으로 확장한다.

### 개발 안정성 원칙

- 기능 변경 전에 화면, API, 타입, 데이터베이스, 외부 연동의 호출 관계를 확인
- 외부 데이터는 내부 도메인 모델로 변환하고 런타임 검증 후 사용
- 공유 타입과 검증 규칙을 단일 기준으로 관리
- 정상 상태뿐 아니라 로딩·빈 결과·오류·권한 거부 상태를 함께 구현
- 데이터 스키마 변경은 마이그레이션과 롤백 방법을 포함
- 변경 후 타입 검사, 자동 테스트, 프로덕션 빌드, 핵심 사용자 흐름을 검증
- 반복되거나 누락 위험이 큰 작업은 검증 스크립트 또는 전용 스킬로 자동화

세부 작업 규칙과 완료 조건은 `docs/development-workflow.md`를 따른다.

## 8. 단계별 실행 계획

### 0단계 — 방향 설정과 조사

- 제품 목표, 경쟁 서비스, 차별화 원칙 확정
- MVP와 제외 범위 확정
- 성공 지표 정의

완료 기준: 이 문서의 제품 방향에 합의

### 1단계 — UX와 디자인 설계

- 사용자 흐름 정의
- 정보 구조와 화면 목록 작성
- 홈, 지도, 상세 화면 와이어프레임 제작
- 색상·타이포그래피·카드·배지 디자인 시스템 확정

완료 기준: 모바일 핵심 화면 프로토타입으로 탐색 흐름 확인

### 2단계 — 데이터 모델과 초기 데이터

- 빵집, 지점, 메뉴, SNS 계정, 출처, 검증 이력, 임시 일정, 제보 스키마 설계
- 대표 빵집 50~100곳의 시드 데이터 구축
- 공식 SNS 계정 식별과 원문 링크 등록
- 중복 지점과 폐점 여부 검수

완료 기준: 핵심 필드 95% 이상 채움, 모든 레코드에 출처와 확인일 존재

### 3단계 — MVP 개발

- 홈, 검색, 지도/목록, 상세, 저장 구현
- 제보와 관리자 검수 흐름 구현
- 모바일 반응형 및 접근성 적용
- 공통 타입·런타임 검증·오류 처리 적용
- 핵심 로직 단위 테스트와 화면/API 통합 테스트 작성

완료 기준: 핵심 사용자 여정을 처음부터 끝까지 수행할 수 있고 타입 검사·자동 테스트·프로덕션 빌드가 통과

### 4단계 — 품질 검증

- 주소·좌표·전화·영업시간 표본 검증
- 공식 SNS 최신 공지와 앱 정보의 일치 여부 표본 검증
- 임시휴무 공지가 상시 영업시간을 잘못 변경하지 않는지 검증
- 모바일 기기와 주요 브라우저 테스트
- 로딩·오류·빈 상태·권한 거부 상태 테스트
- 접근성·성능·SEO 검사
- 화면 → API → 데이터 저장소 연결과 주요 회귀 흐름 검사
- 배포 환경 변수와 롤백 절차 확인

완료 기준: 치명적 오류 0건, 핵심 정보 정확도 목표 달성

### 5단계 — 테스트 배포

- Vercel 미리보기/베타 도메인 배포
- 소규모 사용자 테스트
- 탐색 성공률, 저장률, 오류 제보율 수집
- 피드백에 따라 디자인과 데이터 수정

완료 기준: 외부 사용자가 도움 없이 원하는 빵집을 찾고 저장할 수 있음

### 6단계 — 확장 판단

## 9. 초기 성공 지표

- 테스트 공개 시 검증된 빵집 100곳 이상
- 주소·좌표·전화번호 정확도 98% 이상
- 영업시간에 출처와 확인일 표시율 100%
- 사용자가 60초 안에 조건에 맞는 빵집 상세 화면 도달
- 정보 오류 제보 처리 중앙값 48시간 이내
- 핵심 모바일 화면에서 접근성·가독성 문제 없음
