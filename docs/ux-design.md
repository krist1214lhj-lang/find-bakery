# UX·브랜드 설계

작성 기준일: 2026-06-18  
상태: 1차 설계, 구현 전 검토용

## 1. 브랜드 방향

### 작업명

**빵길**

- 의미: 먹고 싶은 빵을 따라 발견하는 길
- 제품 약속: 정확한 정보를 확인하고 안심하고 출발할 수 있는 빵집 탐색
- 사용 예: “빵길에서 찾기”, “내 빵길에 저장”, “오늘의 빵길”

상표, 앱스토어 이름, 도메인 중복 여부를 확인하기 전까지 내부 작업명으로 사용한다.

### 이름 후보

| 후보       | 장점                                 | 주의점                                       |
| ---------- | ------------------------------------ | -------------------------------------------- |
| 빵길       | 짧고 길찾기·여행 의미가 선명함       | 일반 명사 조합이므로 상표 확인 필요          |
| 굽굽지도   | 따뜻하고 친근하며 시각적 개성이 있음 | 정보 신뢰 제품보다 캐주얼하게 느껴질 수 있음 |
| 빵발견     | 기능이 즉시 이해됨                   | 브랜드 고유성이 약함                         |
| 오늘의빵집 | 일상적 추천에 적합                   | 전국 여행·지도 확장성이 다소 약함            |
| 빵신호     | 최신 영업·품절 정보 이미지와 잘 맞음 | 빵집 탐색 의미를 보조 문구로 설명해야 함     |

### 브랜드 성격

- 따뜻하지만 유아적이지 않음
- 감성적이지만 정보가 흐려지지 않음
- 유명세를 과장하지 않고 근거를 설명함
- 사용자의 기대를 부풀리기보다 방문 실패를 줄임

### 핵심 문구

- 브랜드 문장: **믿고 떠나는 전국 빵집 지도**
- 홈 인사: **오늘은 어떤 빵을 만나러 갈까요?**
- 검증 안내: **영업 정보는 2026년 6월 18일 공식 채널에서 확인했어요.**
- 오래된 정보: **확인한 지 오래됐어요. 방문 전 공식 채널을 확인해 주세요.**
- 정보 충돌: **공식 공지와 지도 정보가 달라 확인 중이에요.**

## 2. 디자인 원칙

### 시각 방향

갓 구운 빵의 색과 종이 포장지의 질감을 연상시키되, 화면 자체는 깨끗하고 현대적으로 구성한다.

### 색상 토큰 초안

| 역할      | 이름       | 값        | 용도                  |
| --------- | ---------- | --------- | --------------------- |
| 배경      | Cream 50   | `#FFF9F0` | 앱 기본 배경          |
| 표면      | Flour 0    | `#FFFFFF` | 카드·시트             |
| 주색      | Crust 600  | `#A95122` | 주요 버튼·활성 상태   |
| 진한 주색 | Crust 800  | `#693217` | 제목·강조             |
| 보조색    | Butter 400 | `#F2BE4A` | 태그·일러스트 포인트  |
| 본문      | Cocoa 900  | `#2F241F` | 기본 텍스트           |
| 보조 본문 | Cocoa 600  | `#71645D` | 설명·메타 정보        |
| 성공      | Fresh 700  | `#287A52` | 공식 확인·영업 중     |
| 주의      | Toast 700  | `#9A5A00` | 오래된 정보·변동 가능 |
| 오류      | Berry 700  | `#A63B43` | 폐점 제보·충돌        |
| 경계      | Flour 200  | `#EDE3D8` | 구분선·테두리         |

색상만으로 상태를 구분하지 않고 아이콘과 텍스트를 함께 사용한다. 모든 본문과 버튼은 WCAG AA 대비를 목표로 실제 구현에서 검사한다.

### 타이포그래피

- 한글 UI: Pretendard 또는 시스템 산세리프
- 숫자·시간·가격은 동일 서체의 tabular 숫자 사용
- 제목은 굵기 700, 본문은 400~500
- 감성 표현을 위해 손글씨체를 핵심 정보에 사용하지 않음

### 형태와 이미지

- 카드 모서리: 16px
- 버튼 모서리: 12px 또는 pill
- 터치 영역: 최소 44×44px
- 빵 사진 비율: 목록 4:3, 상세 대표 이미지 16:10
- 검증 배지는 작지만 항상 텍스트로 의미를 읽을 수 있게 구성
- 사진이 없을 때 빵 종류별 단색 일러스트 또는 중립 플레이스홀더 사용

## 3. 정보 구조

```text
빵길
├─ 홈
│  ├─ 통합 검색
│  ├─ 먹고 싶은 빵
│  ├─ 지역별 빵길
│  ├─ 지금 확인된 빵집
│  └─ 테마 컬렉션
├─ 탐색
│  ├─ 지도
│  ├─ 목록
│  ├─ 검색 결과
│  └─ 필터
├─ 빵집 상세
│  ├─ 핵심 방문 정보
│  ├─ 대표 메뉴
│  ├─ 영업시간·임시 일정
│  ├─ 편의 정보
│  ├─ 유명한 이유
│  ├─ 출처·검증 내역
│  └─ 정보 수정 제보
├─ 저장
│  ├─ 가고 싶은 곳
│  └─ 다녀온 곳(후속 범위)
└─ 내 정보
   ├─ 제보 내역
   ├─ 위치·알림 설정
   └─ 서비스 안내

관리자
├─ 검수 대기
├─ 빵집·지점 관리
├─ 출처·SNS 관리
├─ 임시 일정 관리
├─ 사용자 제보
└─ 변경 이력
```

MVP 하단 내비게이션은 `홈`, `탐색`, `저장` 3개로 시작한다. 로그인과 설정은 상단 프로필 메뉴에 둔다. 기능이 늘기 전까지 빈 탭을 만들지 않는다.

## 4. 핵심 사용자 흐름

### 흐름 A — 여행지의 유명 빵집 찾기

1. 홈에서 지역 선택
2. 지역 결과에서 빵 종류·영업 상태 필터
3. 목록 또는 지도에서 빵집 선택
4. 상세에서 대표 메뉴와 검증된 영업 정보 확인
5. 저장 또는 외부 지도 길찾기

성공 조건: 사용자가 60초 안에 방문 후보를 선택하고 최신 확인일을 찾을 수 있다.

### 흐름 B — 먹고 싶은 빵으로 찾기

1. 홈의 `먹고 싶은 빵`에서 종류 선택
2. 현재 위치 허용 또는 지역 직접 선택
3. 거리·검증 상태·영업 중 조건으로 결과 정렬
4. 상세에서 해당 메뉴의 가격과 확인일 확인

위치 권한을 거부해도 지역 직접 선택으로 같은 목적을 달성할 수 있어야 한다.

### 흐름 C — 정보 신뢰도 확인

1. 상세 상단에서 핵심 정보와 검증 요약 확인
2. `출처와 확인 기록` 열기
3. 항목별 등급, 마지막 확인일, 공식 원문 확인
4. 오래되거나 충돌한 정보는 경고와 대체 행동 확인

대체 행동: 전화하기, 공식 SNS 열기, 오류 제보.

### 흐름 D — 오류 제보

1. 상세에서 `정보가 다른가요?` 선택
2. 오류 유형 선택
3. 현재 확인한 내용과 선택적 출처 입력
4. 제출 후 검수 대기 상태 확인

제보 유형: 영업시간, 휴무, 폐점, 이전, 메뉴·가격, 연락처, 기타.

## 5. 화면별 요구사항

### 홈

- 첫 화면에서 검색, 빵 종류, 지역 중 하나를 즉시 선택 가능
- 검증된 정보가 제품의 강점임을 한 문장으로 전달
- 첫 화면은 빵지순례의 출발표를 연상시키는 편집형 히어로로 구성
- 세로형 브랜드 영상을 장식적 배경이 아닌 탐색의 출발 장면으로 사용
- 출처·확인일·변경 제보를 검색창 바로 아래에서 짧게 설명
- 추천 이유를 `지역 대표`, `공식 정보 최근 확인`, `전문 메뉴`처럼 표시
- 광고와 편집 추천이 생길 경우 명확히 구분

### 탐색

- 지도/목록은 동일한 검색 조건과 결과 집합을 공유
- 화면 전환 시 필터와 스크롤 맥락 유지
- 지도 마커 선택 시 하단 미리보기 카드 표시
- 결과가 없으면 필터 초기화와 인접 지역 탐색 제안
- 위치 권한 거부 시 오류가 아닌 지역 선택 화면 표시

### 상세

정보 우선순위:

1. 영업 여부와 오늘 일정
2. 지점명과 주소
3. 검증 요약과 확인일
4. 대표 메뉴와 가격
5. 전화·길찾기·저장
6. 전체 영업시간과 임시 일정
7. 편의 정보와 유명한 이유
8. 출처·변경 기록

`영업 중`은 기본 영업시간만으로 단정하지 않고 임시휴무·특별 일정까지 계산할 수 있을 때 표시한다.

### 저장

- 로그인 전에는 기기 저장을 허용
- 로그인 후 계정 동기화 선택 제공
- 저장 실패 시 낙관적 UI를 되돌리고 이유 표시
- 삭제보다 저장 해제를 사용해 사용자의 의도를 명확히 함

### 관리자 검수

- 원본 값, 제안 값, 출처, 지점, 적용 기간을 한 화면에서 비교
- 승인·반려·보류와 사유 기록
- 영구 변경과 임시 일정 변경을 분리
- 승인 전 사용자 화면에 확정 정보로 노출하지 않음

## 6. 접근성과 신뢰 UX

- 지도만으로 정보를 전달하지 않고 동등한 목록 제공
- 이미지에 대체 텍스트 제공
- 영업 상태·검증 상태를 색상만으로 구분하지 않음
- 외부 링크는 목적지와 새 화면 열림 여부를 설명
- 위치 권한은 기능 사용 시점에만 요청하고 거부 대안을 제공
- 정확하지 않은 추정값을 사실처럼 표현하지 않음
- AI 요약이 도입되면 출처와 생성 사실을 명시하고 원문으로 이동 가능하게 함

## 7. 분석 이벤트 초안

개인정보를 최소화하며 제품 성공 지표에 필요한 이벤트만 수집한다.

| 이벤트                 | 목적                           |
| ---------------------- | ------------------------------ |
| `search_submitted`     | 검색 사용률과 검색어 유형 파악 |
| `filter_applied`       | 유용한 필터 파악               |
| `bakery_opened`        | 상세 도달률 측정               |
| `verification_opened`  | 검증 정보 사용성 측정          |
| `bakery_saved`         | 방문 의도 측정                 |
| `directions_opened`    | 실제 방문 전환의 대리 지표     |
| `correction_submitted` | 정보 오류와 검수 수요 측정     |

정확한 위치 좌표와 검색 원문은 분석 도구로 보내지 않는 것을 기본으로 한다.

## 8. 구현 연결 기준

| UX 개념          | 데이터/로직 기준                     |
| ---------------- | ------------------------------------ |
| 공식 확인 배지   | 항목별 최신 검증 기록과 공식 출처    |
| 영업 중          | 기본 영업시간 + 임시 일정 + 시간대   |
| 대표 메뉴 가격   | 메뉴 가격의 확인일과 출처            |
| 지도/목록 동기화 | 하나의 검색 쿼리 상태와 결과 ID 집합 |
| 정보가 오래됨    | 항목별 재검증 기한 초과              |
| 지도와 다름      | 활성 출처 간 충돌 상태               |
| 유명한 이유      | 근거 종류와 근거 출처                |

화면 문구가 데이터로 증명되지 않으면 표시하지 않는다.
