# ADR 0001: Supabase와 로컬 프로토타입의 경계

날짜: 2026-06-18  
상태: 채택

## 배경

프로토타입은 Supabase 자격 증명이 없는 환경에서도 저장·제보 UX를 테스트해야 한다. 하지만 화면 컴포넌트가 직접 Supabase와 localStorage를 각각 다루면 서버 연결 시 코드가 분기되고 오류 가능성이 커진다.

## 결정

- 화면은 제보 저장소 함수만 호출한다.
- `/api/reports`가 서버 저장 가능 여부를 판단한다.
- Supabase가 설정되면 서버에 저장한다.
- 설정되지 않았거나 일시적인 서버 장애가 발생하면 검증된 제보를 로컬 큐에 보관한다.
- 4xx 입력 오류는 로컬 폴백으로 숨기지 않고 사용자에게 반환한다.
- 운영 관리자 변경은 RLS를 우회하는 직접 UPDATE가 아니라 검수 RPC 한 경로만 사용한다.
- 로컬 관리자 화면은 개발 또는 명시적 데모 환경에서만 노출한다.

## 결과

장점:

- 데이터베이스 연결 전후에 동일한 제보 화면 유지
- 네트워크 장애에서 사용자 입력 손실 감소
- 검수 상태와 감사 이력의 원자적 변경
- 서버 비밀키의 브라우저 노출 방지

비용:

- 서버 연결 전 로컬 제보는 다른 기기와 공유되지 않음
- 서버 연결 후 로컬 대기 제보를 업로드하는 동기화 작업이 필요
- 실제 인증·RLS 검증은 Supabase 테스트 프로젝트가 필요
