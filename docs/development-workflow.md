# 개발 운영 및 품질 계획

작성 기준일: 2026-06-18

## 1. 목적

기능을 빠르게 추가하는 것보다 변경이 앱 전체에서 올바르게 연결되고, 기존 기능을 깨뜨리지 않으며, 정보 정확성 원칙을 유지하는 것을 우선한다.

## 2. 변경 영향 범위 확인

코드를 수정하기 전에 다음 연결을 추적한다.

```text
사용자 화면
  → 화면 상태와 폼
  → 서비스/쿼리 계층
  → API 또는 서버 액션
  → 도메인 타입과 검증
  → 데이터베이스/외부 API
  → 응답 변환
  → 화면의 성공·빈 상태·오류 상태
```

각 변경에서 아래 항목 중 영향을 받는 것을 기록한다.

- 라우트와 내비게이션
- 컴포넌트 속성과 공유 상태
- 요청·응답 타입
- 데이터베이스 스키마와 마이그레이션
- 캐시와 재검증 정책
- 인증·권한
- 외부 지도·SNS·공공데이터 연동
- 환경 변수
- 분석 이벤트
- 테스트와 문서

## 3. 단일 기준 원칙

- 빵집, 지점, 메뉴, 출처, 검증 상태는 공통 도메인 타입을 사용한다.
- 데이터베이스 스키마와 애플리케이션 타입 간 차이를 자동 검사한다.
- 지도 API와 공공데이터의 원본 구조를 화면에서 직접 사용하지 않는다.
- 날짜와 시간은 저장 형식, 시간대, 화면 표시 규칙을 통일한다.
- 검증 등급과 상태 문구는 한 모듈에서 정의한다.

## 4. 외부 데이터 방어

외부 API와 SNS 정보는 항상 누락되거나 바뀔 수 있다고 가정한다.

- 런타임 스키마 검증 후 내부 모델로 변환
- 타임아웃과 재시도 횟수 제한
- 호출 실패 시 마지막 검증 데이터와 확인 시점 표시
- 출처 충돌 시 자동 덮어쓰기 금지
- API 키는 서버에서만 사용하고 저장소에 커밋하지 않음
- 서비스 약관, 호출량 제한, 데이터 저장 가능 범위 확인
- SNS 원문 삭제나 비공개 전환에 대비해 검증 이력 유지

## 5. 완료 조건

기능은 다음 조건을 모두 만족해야 완료된 것으로 본다.

- 요구된 사용자 흐름이 동작함
- 호출 관계와 타입 검사가 통과함
- 정상·로딩·빈 상태·오류 상태가 구현됨
- 관련 자동 테스트가 존재하고 통과함
- 모바일 화면을 실제 브라우저에서 확인함
- 접근성 기본 항목을 확인함
- 데이터 변경이면 출처와 검증 이력을 확인함
- 환경 변수와 배포 설정을 확인함
- 영향받은 문서를 갱신함

## 6. 테스트 계층

### 단위 테스트

- 외부 데이터 변환
- SNS 상대 날짜의 절대 날짜 변환
- 영업시간과 임시휴무 병합
- 검증 등급 계산
- 검색·정렬·필터

### 통합 테스트

- 검색 요청과 결과 표시
- 상세 정보와 출처 표시
- 저장과 해제
- 사용자 제보와 관리자 검수 상태 변경
- 데이터베이스 권한 정책

### 브라우저 흐름 테스트

- 홈에서 지역 또는 메뉴로 빵집 발견
- 지도와 목록 전환
- 상세에서 공식 출처와 확인일 확인
- 빵집 저장
- 오류 정보 제보
- 외부 지도 길찾기 연결

### 회귀 테스트

수정한 기능뿐 아니라 연결된 핵심 흐름을 최소 한 번 다시 확인한다. 데이터 모델, 공통 타입, 라우팅, 인증을 변경한 경우 전체 핵심 흐름을 검사한다.

## 7. 데이터 변경 안전장치

- 스키마 변경은 마이그레이션 파일로 관리
- 파괴적 변경 전에 기존 데이터 변환 계획 작성
- 초기 데이터와 테스트 데이터를 분리
- 운영 데이터 수정은 변경 이력과 수정 주체 기록
- 폐점·이전 정보는 삭제 대신 상태 변경을 우선
- 임시휴무는 상시 영업시간과 별도 테이블/구조로 관리
- 롤백 가능한 변경 단위 유지

## 8. 배포 게이트

테스트 배포 전 최소 조건:

1. 포맷·린트·타입 검사 통과
2. 단위·통합 테스트 통과
3. 프로덕션 빌드 성공
4. 필수 환경 변수 목록 확인
5. 모바일 핵심 흐름 브라우저 확인
6. 오류 로그와 사용자 제보 경로 확인
7. 데이터 표본 정확성 검사 통과
8. 이전 배포로 되돌리는 방법 확인

현재 로컬 통합 검증 명령:

```bash
npm run verify
```

이 명령은 문서 참조 무결성, 린트, TypeScript 타입 검사, 단위 테스트, 프로덕션 빌드를 순서대로 실행하고 하나라도 실패하면 즉시 중단한다.

배포 환경 변수 확인:

```bash
npm run verify:deploy-env
```

Supabase 로컬/원격 적용 검증 절차는 `docs/database-setup.md`를 따른다.

## 9. 스킬과 자동화 생성 기준

다음 상황에서는 전용 스킬이나 스크립트를 만든다.

- 공식 SNS·지도·공공데이터를 같은 규칙으로 반복 검증할 때
- 빵집 시드 데이터의 필수 필드와 중복을 반복 검사할 때
- 로컬·미리보기·배포 환경을 같은 절차로 반복 확인할 때
- 동일한 브라우저 사용자 흐름을 반복 테스트할 때
- 사람이 수행하면 누락 가능성이 높은 릴리스 점검이 생길 때

자동화 결과만 믿지 않고, 자동화가 실제 요구사항을 검사하는지도 함께 검토한다.
