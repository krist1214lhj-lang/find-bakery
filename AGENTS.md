# 작업 원칙

이 저장소에서는 제품 정확성, 코드 연결성, 회귀 방지를 같은 수준의 필수 조건으로 취급한다.

## 작업 전

1. 요청과 `docs/product-plan.md`의 범위를 확인한다.
2. 변경 대상 파일뿐 아니라 호출자, 피호출자, 타입, 데이터 스키마, 환경 변수, 라우트까지 검색한다.
3. 기존 사용자 변경을 보존하고 관련 없는 파일은 수정하지 않는다.
4. 불명확한 부분은 안전한 가정을 문서화하되, 제품 방향을 바꿀 정도의 결정은 사용자에게 확인한다.

## 구현 중

1. 화면, API, 데이터베이스를 따로 고치지 말고 데이터 흐름 전체를 함께 본다.
2. 공유 타입과 스키마를 단일 기준으로 두고 같은 구조를 여러 파일에 중복 정의하지 않는다.
3. 외부 API 응답은 내부 도메인 모델로 변환한 뒤 사용한다.
4. 로딩, 빈 결과, 오류, 권한 거부, 데이터 누락 상태를 정상 흐름과 함께 구현한다.
5. 빵집 정보에는 출처, 확인 시점, 검증 상태가 누락되지 않게 한다.
6. 새 의존성은 기존 도구로 해결할 수 없는 경우에만 추가한다.
7. 반복되는 작업이 세 번 이상 나타나면 스크립트, 공통 모듈 또는 전용 스킬 후보로 검토한다.

## 변경 후

변경 위험에 맞춰 다음 항목을 실행하고 결과를 확인한다.

기본 통합 검증은 `npm run verify`로 실행한다.

1. 정적 검사: 포맷, 린트, TypeScript 타입 검사
2. 단위 검사: 변환기, 검증 규칙, 날짜 처리, 검색·필터 로직
3. 통합 검사: 화면 → API → 데이터 저장소의 연결
4. 사용자 흐름 검사: 홈 → 검색/지도 → 상세 → 저장/제보
5. 데이터 검사: 출처·확인일·지점 구분·임시 일정 보존
6. 배포 검사: 환경 변수, 빌드, 미리보기 배포, 주요 화면 확인

검사를 실행하지 못했다면 완료로 표현하지 말고 실행하지 못한 이유와 남은 위험을 기록한다.
